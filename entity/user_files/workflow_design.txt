Here’s the pseudocode for each workflow function (processor) grouped by entities, along with an explanation of how they will work and how the functions will be launched.

### 1. User Environment Entity Processors

#### 1.1 Processor: `save_environment`

**Pseudocode:**
```plaintext
function save_environment(meta, environment_data):
    validate environment_data
    # Create or update the environment configuration in the database
    environment_id = database.save_environment(environment_data)
    # Trigger workflow for environment setup
    workflow.trigger("environment_workflow", environment_id)
    return environment_id
```

**Launch Process:**
When a user sends a POST request to `/deploy/cyoda-env`, the environment entity is created or updated, and the `save_environment` function is called asynchronously in the background. The function will validate the incoming data and save it in the database. Once the entity is saved, it triggers the corresponding workflow for further processing.

### 2. Deployment Entity Processors

#### 2.1 Processor: `save_deployment`

**Pseudocode:**
```plaintext
function save_deployment(meta, deployment_data):
    validate deployment_data
    # Create a new deployment request in the database
    deployment_id = database.save_deployment(deployment_data)
    # Trigger workflow for deployment
    workflow.trigger("deployment_workflow", deployment_id)
    return deployment_id
```

**Launch Process:**
When a user sends a POST request to `/deploy/user_app`, the deployment entity is created, and the `save_deployment` function is invoked asynchronously. This function validates the request data, saves the deployment information, and then triggers the relevant workflow for the deployment process.

#### 2.2 Processor: `update_deployment`

**Pseudocode:**
```plaintext
function update_deployment(meta, deployment_data):
    validate deployment_data
    # Update the deployment information in the database
    database.update_deployment(deployment_data.id, deployment_data)
    return deployment_data.id
```

**Launch Process:**
When a user sends a PUT request to `/deploy/user_app/{id}`, this function will be called to update the existing deployment's information. It validates the input and updates the record in the database.

### 3. Environment Status Entity Processors

#### 3.1 Processor: `fetch_environment_status`

**Pseudocode:**
```plaintext
function fetch_environment_status(meta, deployment_id):
    # Retrieve environment status by deployment ID
    status = database.get_environment_status(deployment_id)
    return status
```

**Launch Process:**
When a user sends a GET request to `/deploy/cyoda-env/status/{id}`, this function will be invoked to fetch the status of the specified environment. It interacts with the database to retrieve the relevant information.

### 4. Cancellation Entity Processors

#### 4.1 Processor: `cancel_deployment`

**Pseudocode:**
```plaintext
function cancel_deployment(meta, deployment_id):
    # Validate if the deployment can be canceled
    if can_cancel(deployment_id):
        database.cancel_deployment(deployment_id)
        return "Deployment canceled"
    else:
        return "Cancellation not allowed"
```

**Launch Process:**
When a user sends a POST request to `/deploy/cancel/user_app/{id}`, the function is triggered to process the cancellation of the specified deployment. It checks whether the deployment can be canceled and performs the action accordingly.

### Summary of How Functions Will Work

- Each processor function is designed to handle specific business logic associated with its respective entity.
- Functions will be launched based on user interactions with the defined API endpoints. For example, when a user sends a POST request to create a new environment, the `save_environment` function is triggered.
- Once invoked, the processor functions perform their tasks independently, such as validating data, saving entities, and triggering workflows, without direct knowledge of other processors. This approach maintains separation of concerns and ensures that each function is self-contained.
- The orchestration between the entities and workflows is managed by Cyoda, which ensures that proper events are triggered, and gRPC notifications are sent to the client as needed. This allows for seamless communication and processing of requests throughout the application.
